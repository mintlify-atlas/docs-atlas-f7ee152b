## The Invariant

**WAL ORDERING INVARIANT**: Every governance decision has a durable audit record written to disk (via `fsync`) **before** the decision is returned to the caller. Execution must never precede the audit record — under any failure mode.

```go
// From internal/core/pipeline.go:344-368
// decide writes the WAL record and returns the Decision.
// This is the WAL ORDERING INVARIANT implementation:
// no decision is returned until the record is fsynced.
func (p *Pipeline) decide(...) Decision {
    // ... build record ...
    
    // [9] WAL write — fsync before returning.
    if err := p.wal.Write(rec); err != nil {
        // If we can't write the audit record, we must deny.
        // Execution must never precede the audit record.
        return Decision{
            Effect:     EffectDeny,
            ReasonCode: reasons.WALWriteFailure,
            Reason:     "audit record write failed; denying to preserve WAL invariant",
        }
    }
    
    // [10] Async: replicate to SQLite, update session history
    // [11] Return Decision
    return d
}
```

## Why This Matters

In production AI agent systems, the following scenarios are unacceptable:

1. **Silent execution**: An agent executes a high-value action (e.g., `stripe/refund $10,000`) but the audit record is lost due to a crash → no evidence the action was governed
2. **Regulatory gap**: A HIPAA-covered system processes PHI, but the access log is incomplete due to write buffering → compliance violation
3. **Forensic blind spot**: A security incident occurs, but the DPR chain has gaps → impossible to reconstruct the attack timeline

The WAL-first invariant **guarantees** that if an action is permitted, an audit record exists on disk. No gaps are possible.

## Implementation Details

### 11-Step Evaluation Pipeline

From the README (`/home/daytona/workspace/source/README.md:228-251`):

```
CanonicalActionRequest
 │
 ├─[1]  Kill switch check      atomic.Bool per agent_id — nanoseconds
 ├─[2]  Phase check            tool visible in current workflow phase?
 ├─[3]  Pre-execution scanners shell, secret, PII — parallel goroutines
 ├─[4]  Session state read     sync.Map counters + ring buffer
 ├─[5]  History ring read      last N calls within T seconds
 ├─[6]  External selector fetch lazy, parallel, cached, timeout-bounded
 ├─[7]  Policy evaluation      expr-lang bytecode, first-match-wins, ~1μs/rule
 ├─[8]  Decision               PERMIT | DENY | DEFER | SHADOW
 ├─[9]  WAL write ──────────── fsync to local disk BEFORE returning decision
 ├─[10] Async                  replicate to SQLite DPR, update session + history
 └─[11] Return Decision        to adapter
```

**Critical ordering**: Step 9 (WAL write) happens **before** step 11 (return). If step 9 fails, the function returns `DENY` immediately — execution is blocked.

Step 10 (SQLite replication) happens **asynchronously** because SQLite is for queries, not durability. The WAL is the source of truth.

### WAL Write Implementation

From `/home/daytona/workspace/source/internal/core/dpr/wal.go:37-56`:

```go
// Write appends a record to the WAL and calls fsync before returning.
// This must be called before the decision is delivered to the adapter.
func (w *WAL) Write(rec *Record) error {
    w.mu.Lock()
    defer w.mu.Unlock()

    line, err := json.Marshal(rec)
    if err != nil {
        return fmt.Errorf("marshal DPR record: %w", err)
    }
    line = append(line, '\n')

    if _, err := w.file.Write(line); err != nil {
        return fmt.Errorf("write WAL: %w", err)
    }
    if err := w.file.Sync(); err != nil {
        return fmt.Errorf("fsync WAL: %w", err)
    }
    return nil
}
```

**Key details**:
- `w.mu.Lock()`: Serializes writes — no concurrent WAL corruption
- `w.file.Write()`: Appends JSON line to the file
- `w.file.Sync()`: **Blocks until the OS flushes buffers to disk** (fsync syscall)
- **No early return**: If any step fails, the entire function fails and returns an error

### Failure Modes

| Failure Scenario | Behavior | Audit Gap? |
|-----------------|----------|------------|
| **Process crash before WAL write** | Decision never returned to adapter → no execution | ✅ No gap |
| **Process crash during WAL write** | Decision never returned → no execution | ✅ No gap |
| **fsync fails** (disk full, I/O error) | `Write()` returns error → `decide()` returns DENY → no execution | ✅ No gap |
| **Process crash after WAL write, before SQLite** | WAL contains record, SQLite does not → WAL replay on restart populates SQLite | ✅ No gap |
| **Power loss after fsync** | Record is on disk (fsync guarantee) → survives reboot | ✅ No gap |
| **Filesystem corruption** | WAL may be unreadable → restore from backup, verify chain | ⚠️ Data loss (not audit gap) |

<Warning>
**Filesystem Requirement**: The WAL-first invariant depends on the filesystem honoring fsync semantics. Use ext4, XFS, or ZFS. **Avoid NFS without `sync` mounts** — async NFS can lose fsynced data on server crashes.
</Warning>

### WAL Replay on Restart

When the daemon restarts, it seeds the in-memory chain hash cache from the SQLite store (`pipeline.go:93-102`):

```go
// Seed chain hashes from SQLite so the DPR chain is continuous across restarts.
if cfg.Store != nil {
    if agents, err := cfg.Store.KnownAgents(); err == nil {
        for _, agentID := range agents {
            if h, err := cfg.Store.LastHash(agentID); err == nil && h != "" {
                p.chainMu[agentID] = h
            }
        }
    }
}
```

If the SQLite store is missing records that exist in the WAL (e.g., due to a crash), the WAL can be replayed:

```bash
# Manual WAL replay (future CLI command)
faramesh audit replay ~/.faramesh/wal.jsonl --store ~/.faramesh/faramesh.db
```

## Performance Impact

fsync is **expensive** — it forces a round-trip to the disk controller. On modern SSDs:
- **NVMe SSD**: ~100 μs per fsync
- **SATA SSD**: ~500 μs per fsync
- **HDD**: ~5-10 ms per fsync (due to rotational latency)

Faramesh targets **sub-millisecond governance latency** for most decisions:
- Kill switch check: **nanoseconds** (atomic read)
- Pre-execution scanners: **~100 μs** (parallel regex)
- Policy evaluation: **~1 μs per rule** (compiled bytecode)
- **WAL write (fsync): ~100-500 μs** (dominates latency on SSD)

Total latency on NVMe: typically **< 1 ms**. From the demo output:
```
[10:00:15] PERMIT  get_exchange_rate      latency=11ms  ← includes network selector fetch
[10:00:17] DENY    shell/run              latency=0ms   ← scanner-only, no fsync
[10:00:18] PERMIT  read_customer          latency=9ms
```

### Batch Optimization (Future)

For ultra-high-throughput workloads (>10,000 decisions/sec), consider:
- **Group commit**: Batch multiple WAL writes into a single fsync
- **Async adapter**: Return a `Future<Decision>` and unblock the agent while fsync completes

These optimizations **must preserve the invariant**: execution cannot proceed until fsync completes.

## Related Guarantees

### 1. Atomic Decision

Each `Evaluate()` call is **atomic**: either a DPR record is created and the decision is returned, or an error is returned and no record exists. No partial states.

### 2. Chain Continuity

Each DPR record links to the previous record's hash. If the daemon crashes and restarts, the chain continues from the last known hash — no forks, no gaps.

### 3. Timestamp Monotonicity

Records are timestamped with `req.Timestamp` (set by the adapter) or `time.Now()` (set by the pipeline). Timestamps are **not guaranteed** to be monotonically increasing (due to clock skew), but the `prev_record_hash` chain defines the canonical order.

## FAQ

### What if SQLite replication fails?

SQLite is for **queries**, not durability. If SQLite replication fails (async goroutine in step 10), the WAL still contains the record. You can:
- Query the WAL directly: `cat ~/.faramesh/wal.jsonl | jq`
- Replay the WAL into SQLite: `faramesh audit replay`

The governance decision is **not affected** — execution proceeds based on the WAL write, not SQLite.

### Can I disable fsync for development?

No. Faramesh does not expose an option to disable fsync because it would **violate the invariant**. For development, use:
- **In-memory mode**: `faramesh demo` uses `NullWAL` (no fsync, no persistence)
- **Fast disk**: NVMe SSDs make fsync fast enough for production

### What about distributed systems?

In a multi-node deployment:
- Each node writes its own local WAL (fsync to local disk)
- Records are replicated to a central PostgreSQL or Horizon Cloud Platform
- The **local WAL** is still the source of truth for that node's decisions

See [Compliance Exports](/security/compliance-exports) for multi-node audit aggregation.
